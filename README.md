# Call Analyzer — аналіз клієнтських дзвінків СТО

Бот для автоматичного аналізу якості телефонних дзвінків автосервісу.
Забирає аудіозаписи з Google Drive, транскрибує мову в текст, аналізує якість
обслуговування за скриптом менеджера та записує результати в Google Sheets.

## Як це працює

```
Google Drive (аудіо .mp3)
        │
        ▼
   faster-whisper (транскрибація, модель large-v3)
        │
        ▼
   Ollama / Qwen 2.5 7B (аналіз за 22 критеріями)
        │
        ▼
   Google Sheets (звіт з оцінками та коментарями)
```

Pipeline обробляє кожен файл послідовно: завантаження → транскрибація → LLM-аналіз → запис у таблицю.
Транскрипти кешуються локально в `downloads/` — при повторному запуску Whisper не перетранскрибує вже оброблені файли.


## Вимоги

- Python 3.10+
- Linux (або WSL на Windows)
- [Ollama](https://ollama.ai/) з моделлю `qwen2.5:7b`
- Google Cloud Service Account з доступом до Drive та Sheets API
- 8+ GB RAM (для Whisper large-v3 на CPU)


## Встановлення

### 1. Клонувати репозиторій

```bash
git clone git@github.com:Gennadiy-Zubariev/call-analyzer.git
cd call-analyzer
```

### 2. Створити віртуальне середовище

```bash
python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
```

### 3. Налаштувати Ollama

```bash
# Встановити Ollama: https://ollama.ai/download
ollama pull qwen2.5:7b
ollama serve  # запустити сервер (якщо не запущений)
```

### 4. Налаштувати Google Cloud

1. Створити Service Account у Google Cloud Console
2. Увімкнути Google Drive API та Google Sheets API
3. Завантажити JSON-ключ у `credentials/service-account.json`
4. Створити Google Sheets таблицю вручну та поділити з Service Account як Editor
5. Створити папки `call-analyzer-input` та `call-analyzer-work` у Google Drive,
   поділити з Service Account

> **Важливо:** Service Account з безкоштовного тарифу не має Storage Quota на Drive.
> Тому таблиця створюється вручну, а транскрипти зберігаються тільки локально.

### 5. Заповнити .env

```bash
cp .env.example .env
# Відредагувати .env — заповнити ID папок та таблиці
```

Необхідні змінні:

| Змінна | Опис |
|--------|------|
| `GOOGLE_SERVICE_ACCOUNT_FILE` | Шлях до JSON-ключа service account |
| `SOURCE_DRIVE_FOLDER_ID` | ID папки з вхідними аудіозаписами |
| `WORK_DRIVE_FOLDER_ID` | ID робочої папки |
| `WORK_SHEET_ID` | ID Google Sheets таблиці |
| `WHISPER_MODEL` | Модель Whisper (рекомендовано `large-v3`) |
| `WHISPER_DEVICE` | `cpu` або `auto` (GPU потребує CUDA + cuBLAS) |
| `OLLAMA_HOST` | Адреса Ollama сервера (`http://localhost:11434`) |
| `OLLAMA_MODEL` | Модель для аналізу (`qwen2.5:7b`) |


## Запуск

```bash
python3 main.py
```

Pipeline знайде всі аудіофайли у вхідній папці Google Drive, обробить кожен
та запише результати в таблицю. Прогрес відображається в консолі.

Орієнтовний час: 5-7 хвилин на файл (CPU), основне — транскрибація Whisper.
При повторних запусках транскрипти беруться з кешу, тому аналіз займає ~15 с/файл.

## Структура таблиці

Таблиця містить 22 колонки, згруповані у блоки:

- **Ідентифікація** — дата, транскрибація, тип звернення, номер телефону, філія, менеджер
- **Скрипт** — привітання та представлення
- **Інформація про автомобіль** — кузов, рік, пробіг, діагностика
- **ДОП продажі** — пропозиція комплексної діагностики, попередні роботи
- **Скрипт** — запис на сервіс, завершення розмови
- **Інфо** — тип роботи зі списку ТОП-100
- **Результат** — оцінка, запчастини, коментар, сума балів


## Архітектурні рішення

### Strategy Pattern для транскрибації

`transcriber.py` реалізує абстрактний базовий клас `Transcriber` з двома
конкретними реалізаціями: `LocalWhisperTranscriber` (faster-whisper локально)
та `OpenAIWhisperTranscriber` (через OpenAI API). Фабрична функція `make_transcriber()`
обирає реалізацію на основі конфігурації. Додати нового провайдера — це один клас
без зміни решти коду.

### Ідемпотентність

- Транскрипти кешуються в `downloads/` — Whisper не перетранскрибує існуючі файли
- `ensure_headers()` у Sheets перевіряє шапку перед записом
- `file_in_folder()` у Drive перевіряє наявність файлу

### Fail-safe аналіз

Якщо LLM не зміг повернути валідний JSON за 3 спроби, `analyzer.py` повертає
fallback-об'єкт з `is_problematic=True` та поясненням. Pipeline не зупиняється
через одну помилку аналізу.

### Нормалізація типів робіт

LLM може повернути довільний текст замість точної назви зі списку ТОП-100.
`_normalize_work_type()` знаходить найближчу відповідність за ключовими словами,
або повертає "інший варіант" якщо збіг недостатній.


## Якість аналізу та обмеження

### Поточна модель: Qwen 2.5 7B (безкоштовна, локальна)

Модель працює прийнятно для базового аналізу:

**Що розпізнає добре:**
- Привітання та представлення менеджера
- Марку/модель/рік автомобіля (навіть якщо клієнт сам озвучив)
- Факт пропозиції діагностики з ціною
- Запис на сервіс (дата/час)
- Конфліктні/проблемні дзвінки
- Загальний тип роботи

**Відомі обмеження 7B моделі:**
- Ім'я менеджера іноді плутається з ім'ям клієнта
- Тип роботи може бути неточним (наприклад, "Діагностика двигуна" замість
  "Комп'ютерна діагностика")
- Оцінка `score` та `followed_top100_instructions` суб'єктивні та варіюються
  між запусками (temperature=0.1, але повна детермінованість неможлива)
- Складні дзвінки (конфлікти, follow-up після ремонту) аналізуються менш точно

### Рекомендації для production

Архітектура проекту вже підтримує заміну LLM-бекенду без зміни решти коду.
Для підвищення якості аналізу рекомендовано:

1. **Більша локальна модель** — `qwen2.5:14b` або `qwen2.5:32b` через Ollama.
   Потребує 16-32 GB RAM, але значно краще розуміє контекст розмови.

2. **Хмарний API** — GPT-4o, Claude або Gemini через API. Найкраща якість,
   вартість ~$0.01-0.05 за дзвінок. Strategy Pattern дозволяє додати нову
   реалізацію `Analyzer` без зміни pipeline.

3. **Безкоштовні хмарні варіанти** — Groq (Llama 3.3 70B, безкоштовний tier),
   Google Gemini (безкоштовний tier). Якість значно краща за локальну 7B модель.

Перехід потребує лише нового класу-аналізатора та змінної в `.env`.
